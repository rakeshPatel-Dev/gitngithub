Section 01

What Is Version Control?

Imagine you're playing a video game. Before a tough boss fight, you save a checkpoint. If things go wrong, you reload that checkpoint instead of starting over. Version control works the same way — it records changes to your files over time, letting you rewind to any previous version whenever you need to.

Without version control, you might end up with folders like:

# The nightmare we all know: project_v1.py project_v2_final.py project_v2_final_REAL.py project_v3_FINALFINAL_USE_THIS.py

Git eliminates this chaos. It's a distributed version control system that tracks every change, who made it, and when — all on your local machine.

Git vs GitHub — They're Not the Same

GitGitHub
Local tool installed on your computerCloud-based hosting service
Works completely offlineRequires internet to sync
Manages version historyFacilitates collaboration & backup
Command-line or GUI toolsWebsite with social features (PRs, Issues)
Free and open-sourceFree for public/private repos (with limits)
💡 Key Insight Think of Git as the engine and GitHub as the garage. The engine does all the work locally; the garage is where you store, share, and show off your vehicle to the world.

📝 Reflection Task

Before diving into commands, solidify your mental model.

  • Explain version control to a friend using the "video game checkpoint" analogy.
  • Write down 3 differences between Git and GitHub in your own words.
  • Think of a project where version control would have saved you time. Describe it.
Section 02

Pre-Workshop Setup

The most common bottleneck in workshops is installation issues. Complete these steps before the session begins.

1. Create a GitHub Account

Visit github.com and sign up for a free account. Choose a professional username — it'll appear in your resume someday.

2. Install Git

Download Git for Windows from git-scm.com. During installation:

  • Select "Use the nano editor by default" (easier than Vim for beginners)
  • Choose "Git from the command line and also from 3rd-party software"
  • This installs Git Bash — use it for all commands in this workshop

Git is often pre-installed. Open Terminal and type:

git --version

If not installed, you'll be prompted to install Xcode Command Line Tools. Alternatively, install via Homebrew: brew install git.

⚠️ Note for macOS Catalina+ The default shell is now zsh. All commands in this workshop work identically in zsh and bash.

Use your package manager:

# Debian/Ubuntu: sudo apt-get install git # Fedora: sudo dnf install git # Arch: sudo pacman -S git

3. Choose a Text Editor

We recommend Visual Studio Code (free, powerful). Atom and Notepad++ are also fine. Never use Microsoft Word — it inserts hidden formatting that breaks code.

4. Configure Your Identity

Every commit is stamped with your name and email. Set these globally so all your projects use them:

git config --global user.name "Your Name" git config --global user.email "your-email@example.com"

Verify with:

git config --list

✅ Setup Verification Task

Make sure everything is ready before moving on.

  • Create a GitHub account and verify your email.
  • Install Git and run git --version — note the version number.
  • Install VS Code (or another code editor).
  • Run both git config commands and verify with git config --list.
Section 03

The Local Workflow — The "Three States"

Files in Git live in one of three states. Understanding this is the key to mastering Git:

  1. Working Directory — Where you create, edit, and delete files. This is your normal project folder.
  2. Staging Area (Index) — A prep zone where you select which changes go into the next snapshot. Think of it as a "shopping cart" for changes.
  3. Local Repository — The permanent database where committed snapshots are stored safely.
🧠 Mental Model Working Directory → git add → Staging Area → git commit → Repository. You don't have to commit every file at once — the staging area lets you craft meaningful, atomic commits.

Essential Commands

CommandPurpose
git initInitialize a new Git repository in the current folder
git statusShow which files are modified, staged, or untracked
git add <file>Stage a specific file for the next commit
git add .Stage all changes in the current directory
git commit -m "message"Save staged changes as a snapshot with a message
git logView the commit history
git log --onelineCompact one-line-per-commit history
git diffShow unstaged changes line-by-line

Your First Local Repo — Step by Step

# 1. Create a project folder mkdir my-first-repo && cd my-first-repo # 2. Initialize Git git init # 3. Create a file echo "# Hello World" > README.md # 4. Check status (file is 'untracked') git status # 5. Stage the file git add README.md # 6. Check status again (now 'staged') git status # 7. Commit the snapshot git commit -m "Initial commit: add README" # 8. View history git log --oneline

Atomic Commits & Good Messages

An atomic commit is a single, logical change — not a random dump of everything you did that day. Good commit messages follow a simple convention:

  • Use the imperative mood: "Add login feature" (not "Added" or "Adding")
  • Keep the first line under 72 characters
  • Be descriptive but concise: "Fix null pointer in user auth" beats "fix bug"
# ✅ Good commit messages: git commit -m "Fix password reset token expiration" git commit -m "Add unit tests for shopping cart" # ❌ Bad commit messages: git commit -m "stuff" git commit -m "fixed the thing"

🛠️ Hands-On Task: Local Repo

Practice the three-state workflow until it feels natural.

  • Create a new folder, run git init, and create 3 different files.
  • Stage and commit each file individually with meaningful messages.
  • Modify one file, then use git diff to see your changes before staging.
  • Run git log --oneline and verify all 3 commits appear.
  • Make a 4th commit that combines changes to 2 files (stage both, then commit once).
Section 04

Connecting to GitHub — Remotes

Now that you're comfortable locally, let's connect your repository to the cloud. A remote is a pointer to a repository hosted elsewhere (usually GitHub).

Creating a Repository on GitHub

  1. Log in to github.com
  2. Click the "+" icon → "New repository"
  3. Give it a name (e.g., my-first-repo)
  4. Do NOT initialize with README, .gitignore, or license (you already have a local repo)
  5. Click "Create repository"

GitHub will show instructions. You want the "…or push an existing repository from the command line" section.

Linking and Pushing

# Add a remote called 'origin' pointing to your GitHub repo git remote add origin https://github.com/YOUR_USERNAME/my-first-repo.git # Verify the remote git remote -v # Push your commits to GitHub (first time) git push -u origin main

The -u flag sets upstream tracking, so future pushes can be just git push.

Cloning — Starting from an Existing Repo

If someone else already has a repo on GitHub, you can download it:

git clone https://github.com/SOME_USER/some-repo.git cd some-repo

This downloads the entire repository including its full history.

Pulling — Staying in Sync

If changes were made on GitHub (by you or collaborators), pull them down:

git pull

This fetches remote changes and merges them into your local branch.

☁️ Hands-On Task: Push & Clone

Connect your local work to the cloud.

  • Create a new empty repo on GitHub (no README, no .gitignore).
  • Link your local repo using git remote add origin <url>.
  • Push your commits with git push -u origin main.
  • Refresh GitHub in your browser — confirm all files and commits are visible.
  • Clone a popular open-source repo (e.g., github.com/octocat/Hello-World) to a different folder.
Section 05

Branching & Merging

Branches are parallel universes for your code. They let you work on new features, bug fixes, or experiments without touching the stable main branch.

🌿 Branching Philosophy The main branch should always be deployable. Never commit directly to main — create a branch, do your work there, and merge it back when it's ready.

Branch Commands

CommandPurpose
git branchList all local branches
git branch <name>Create a new branch
git switch <name>Switch to an existing branch (modern syntax)
git checkout <name>Switch to a branch (older syntax, still common)
git switch -c <name>Create AND switch to a new branch
git merge <branch>Merge the specified branch into the current one
git branch -d <name>Delete a branch (after merging)

Full Branching Workflow

# Create and switch to a feature branch git switch -c feature/add-login # Make changes, then commit echo "login code here" > login.js git add login.js git commit -m "Add login functionality" # Switch back to main git switch main # Merge the feature branch into main git merge feature/add-login # Delete the feature branch (cleanup) git branch -d feature/add-login

Merge Conflicts — Don't Panic

When two branches change the same part of the same file, Git can't decide which version to keep. This is a merge conflict. Git will mark the conflicting sections:

<<<<<<< HEAD console.log("This is from the main branch"); ======= console.log("This is from the feature branch"); >>>>>>> feature/add-login

Resolve conflicts by editing the file to keep the correct version, removing the markers, then staging and committing.

🌿 Hands-On Task: Branch & Merge

Practice the full branching workflow.

  • Create a branch called feature/experiment and switch to it.
  • Create a new file and commit it on the feature branch.
  • Switch back to main — notice the file disappears (it only exists on the branch).
  • Merge the feature branch into main and confirm the file appears.
  • Delete the feature branch with git branch -d feature/experiment.
  • Bonus: Intentionally create a merge conflict and resolve it.
Section 06

Collaboration — Pull Requests & Forks

Pull Requests (PRs) are the heart of collaboration on GitHub. They let you propose changes, get feedback, and merge code with confidence.

The PR Workflow

  1. Fork a repository (your own copy on GitHub)
  2. Clone your fork locally
  3. Create a branch for your changes
  4. Make commits and push to your fork
  5. Open a Pull Request on GitHub comparing your branch to the original repo
  6. Team reviews, discusses, and eventually merges your code
💡 Pro Tip A PR is a conversation, not just a code dump. Write a clear description, link related issues, and be responsive to feedback. This is how professional software is built.

Fork vs Clone

  • Fork = Your own copy of someone else's repo on GitHub (you can't push to theirs)
  • Clone = A local copy on your computer (of any repo you have access to)

🤝 Hands-On Task: Simulate a PR

Pair up with someone or simulate the workflow solo.

  • Fork a partner's repository (or fork octocat/Hello-World).
  • Clone your fork locally and create a new branch.
  • Make a meaningful change (add a file, fix a typo, etc.) and commit it.
  • Push the branch to your fork on GitHub.
  • Open a Pull Request from your fork's branch to the original repo.
  • Review the PR diff on GitHub — observe how changes are displayed.
Section 07

Best Practices & .gitignore

The .gitignore File

Not everything belongs in version control. A .gitignore file tells Git which files to ignore — system files, dependencies, build artifacts, and sensitive data.

# .gitignore example node_modules/ # Dependencies (reinstall with npm) .env # Environment variables (secrets!) *.log # Log files .DS_Store # macOS system file dist/ # Build output __pycache__/ # Python cache

Create this file early in your project. You can find templates at gitignore.io or in GitHub's template dropdown.

Essential Habits

  • Commit often, push daily — small commits are easier to understand and revert
  • Write descriptive commit messages — future you will thank present you
  • Never commit secrets — API keys, passwords, tokens belong in environment variables
  • Review your changes with git diff before staging
  • Keep main deployable — all tests should pass on the main branch
  • Use branches for every feature, bug fix, or experiment

Conventional Commits (Bonus)

Many teams use a structured format:

git commit -m "feat: add user authentication" git commit -m "fix: resolve null pointer in cart" git commit -m "docs: update API documentation" git commit -m "chore: update dependencies"

🏆 Hands-On Task: Best Practices

Build good habits from day one.

  • Create a .gitignore file in your repo with at least 5 sensible entries.
  • Commit the .gitignore file with the message "chore: add .gitignore".
  • Review your last 3 commit messages — rewrite any that are vague.
  • Try using the conventional commit format for your next 3 commits.
Section 08

Troubleshooting & Undoing Mistakes

Everyone makes mistakes. Git's undo tools are powerful — but they can also be destructive. Learn them carefully.

Undoing Changes — Safety Levels

SituationSafe CommandRisk Level
Unstage a file (keep changes)git restore --staged <file>🟢 Safe
Discard unstaged changesgit restore <file>🟡 Moderate
Undo last commit (keep changes)git reset --soft HEAD~1🟡 Moderate
Undo last commit (discard changes)git reset --hard HEAD~1🔴 Dangerous
Revert a published commitgit revert <commit-hash>🟢 Safe
⚠️ Golden Rule Never use git reset --hard on commits you've already pushed to a shared branch. Use git revert instead — it creates a new commit that undoes the old one, preserving history.

Authentication Issues

GitHub no longer accepts passwords for command-line Git operations (since August 2021). If you see an authentication error, use one of these:

  • Personal Access Token (PAT) — Generate one at GitHub → Settings → Developer Settings → Personal Access Tokens → Tokens (classic). Use it as your "password" when prompted.
  • SSH Keys — More secure, set up once. Generate with ssh-keygen and add the public key to GitHub → Settings → SSH and GPG Keys.
  • Git Credential Manager — Bundled with Git for Windows; handles tokens automatically.

🔧 Hands-On Task: Fixing Mistakes

Practice undoing in a safe environment.

  • Stage a file, then unstage it with git restore --staged <file>.
  • Make an unwanted change, then discard it with git restore <file>.
  • Make a "bad" commit, then undo it with git reset --soft HEAD~1 (changes stay).
  • Create a Personal Access Token on GitHub (save it in a password manager).
  • Use git revert to safely undo a commit while keeping history intact.
Section 09

Advanced Topics (Bonus)

These tools separate comfortable users from power users. Master them gradually.

Stashing — Temporary Storage

Need to switch branches but not ready to commit? Stash your changes:

git stash # Save all changes and revert to clean state git stash pop # Reapply the most recent stash git stash list # View all stashes git stash drop # Delete a specific stash

Rebasing — Clean History

Rebasing rewrites history to make it linear. It's an alternative to merging:

# Rebase your feature branch onto the latest main git switch feature/my-feature git rebase main
⚠️ Caution Never rebase a branch that others are working on. Rebasing rewrites commit hashes, causing confusion for collaborators. Use rebase only on your private branches.

Cherry-Picking

Apply a specific commit from one branch to another:

git cherry-pick <commit-hash>

Interactive Rebase — Squashing

Combine multiple messy commits into one clean commit before merging:

git rebase -i HEAD~4 # Interactively rebase the last 4 commits

Git Aliases — Speed Boost

Create shortcuts for frequent commands:

git config --global alias.co checkout git config --global alias.br branch git config --global alias.st status git config --global alias.lg "log --oneline --graph --all"

🚀 Advanced Task

Level up with power-user techniques.

  • Stash some uncommitted changes, switch branches, then pop the stash.
  • Create a feature branch, make 3 small commits, then use interactive rebase to squash them into 1.
  • Set up at least 3 Git aliases that you'll actually use.
  • Cherry-pick a single commit from one branch to another.
  • Use git log --oneline --graph --all to visualize your repo's history.
